Release verification

Check downloaded release assets before using them.

AnoSecureKit releases publish checksums, GitHub artifact attestations, and SPDX SBOM metadata. Verify assets from the same release before building from source archives or using binary archives.

Open releases Read full guide

Download together

Use assets from one GitHub Release only: `SHA256SUMS.txt`, the archive you plan to use, and the release SBOM if you need metadata.

Check checksums

Compare each downloaded asset against the digest listed in `SHA256SUMS.txt` before unpacking or using it.

Know the limit

Verification shows assets match release workflow outputs. It does not prove external audit status or suitability for high-risk systems.

Checksum verification

On Linux or macOS:

sha256sum -c SHA256SUMS.txt

On Windows PowerShell, compare the printed digest with `SHA256SUMS.txt`:

Get-FileHash .\anosecurekit-0.2.2-source.tar.gz -Algorithm SHA256

GitHub artifact attestations

Use GitHub CLI to verify the checksum file and the asset you downloaded:

gh attestation verify SHA256SUMS.txt --repo anothel/AnoSecureKit-Community
gh attestation verify anosecurekit-0.2.2-source.tar.gz --repo anothel/AnoSecureKit-Community

The attestation should point back to the `anothel/AnoSecureKit-Community` repository and its release workflow.

SBOM metadata

Release SBOM files are named like `anosecurekit-X.Y.Z-release.spdx.json`. Use the SBOM to inspect release asset names and recorded SHA-256 checksums. It is metadata, not a replacement for checksum and attestation verification.

Source or binary archive

  • Use source archives when rebuilding locally, auditing code, or using CMake `FetchContent`.
  • Use binary archives only when the platform and trust model match your deployment.
  • Do not mix source archives, binary archives, and checksum files from different versions.
