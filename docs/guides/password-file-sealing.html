Guide

Password file sealing derives the file key with scrypt.

AnoSecureKit writes password-based file ciphertext as `SKP1`. It uses the same chunked AES-256-GCM layout as raw-key file sealing, but derives the 32-byte file key from caller-provided password bytes with the current default backend's scrypt implementation.

What it is

Password file sealing is chunked authenticated encryption for files where the caller has password bytes instead of a stored 32-byte key.

Password bytes

Password input is used exactly as supplied. AnoSecureKit does not trim, normalize, encode, prompt, store, or read passwords from environment variables.

Path safety

Path-based open has the same no-overwrite and temporary-file commit behavior as raw-key file sealing.

When to use it

  • Use password file sealing when a workflow must be unlocked by user-owned password bytes.
  • Prefer raw-key file sealing when an application can safely manage a 32-byte key file.
  • Use `verify-file-password` when a pipeline needs authentication without plaintext output.
  • Use AAD to bind external context such as record type, tenant ID, or schema version.

CLI example

The CLI reads raw password bytes from `--password-file`. Empty password files are rejected, and the command does not prompt interactively. 

anosecurekit seal-file-password --in plain.bin --out plain.bin.skp --password-file password.bin
anosecurekit verify-file-password --in plain.bin.skp --password-file password.bin
anosecurekit open-file-password --in plain.bin.skp --out restored.bin --password-file password.bin

AAD example

AAD is authenticated but not stored. Opening or verifying requires the exact same AAD bytes used while sealing. 

anosecurekit seal-file-password --in plain.bin --out plain.bin.skp --password-file password.bin --aad-text record:v1
anosecurekit open-file-password --in plain.bin.skp --out restored.bin --password-file password.bin --aad-text record:v1

C++ API example

#include "anosecurekit/file.hpp"

#include <cstddef>

anosecurekit::bytes password{
    std::byte{'c'}, std::byte{'o'}, std::byte{'r'}, std::byte{'r'},
    std::byte{'e'}, std::byte{'c'}, std::byte{'t'}};

anosecurekit::seal_file_with_password("plain.bin", "plain.bin.skp", password);
anosecurekit::verify_file_with_password("plain.bin.skp", password);
anosecurekit::open_file_with_password("plain.bin.skp", "restored.bin", password);

`SKP1` file shape

  • Fixed 1 MiB chunk size.
  • Random per-file scrypt salt.
  • scrypt output size: 32 bytes.
  • AES-256-GCM encryption for each chunk.
  • Chunk nonce made from an 8-byte random file nonce prefix plus a 32-bit chunk index.

Fixed scrypt profile

  • `N = 32768`.
  • `r = 8`.
  • `p = 1`.
  • `maxmem = 64 MiB`.

Unsupported stored scrypt parameters are rejected. Future KDF profiles require the compatibility gate documented in the repository.

Security notes

  • Empty passwords are rejected.
  • Wrong passwords, wrong AAD, malformed structure, truncation, appended data, and tag failures reject the file.
  • Keep password files out of logs, URLs, command history, backups, crash dumps, and source control.
  • Do not trust stdout or stream output from open operations until the process exits successfully.
  • Use a separate security review before using AnoSecureKit in high-risk systems.
