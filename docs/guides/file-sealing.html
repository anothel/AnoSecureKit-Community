Guide

File sealing encrypts files in authenticated chunks.

AnoSecureKit writes raw-key file ciphertext as `SKF1`. It uses a caller supplied 32-byte key, derives a per-file key with HKDF-SHA256, and authenticates every chunk before a path-based restore is committed.

What it is

File sealing is chunked authenticated encryption for files. It is for file workflows that do not fit naturally into one small `SKT1` packet.

Path safety

Path-based open refuses existing outputs, writes a temporary file, and renames it only after the whole sealed file authenticates.

Stream caveat

Stream overloads and CLI `--out -` write to caller-owned output. Treat restored bytes as trusted only after the command or function succeeds.

When to use it

  • Use file sealing for files that need confidentiality and tamper detection.
  • Use `verify-file` when a pipeline needs authentication without plaintext output.
  • Use AAD to bind external context such as record type, tenant ID, or schema version.
  • Use password-based file sealing only when users must provide a password instead of a key file.

CLI example

Generate a raw 32-byte key file, seal the input, verify the sealed file, then restore to a fresh output path. 

anosecurekit keygen --out key.hex
anosecurekit seal-file --in plain.bin --out plain.bin.skf --key-file key.hex
anosecurekit verify-file --in plain.bin.skf --key-file key.hex
anosecurekit open-file --in plain.bin.skf --out restored.bin --key-file key.hex

AAD example

AAD is authenticated but not stored. Opening or verifying requires the exact same AAD bytes used while sealing. 

anosecurekit seal-file --in plain.bin --out plain.bin.skf --key-file key.hex --aad-text record:v1
anosecurekit open-file --in plain.bin.skf --out restored.bin --key-file key.hex --aad-text record:v1

C++ API example

#include "anosecurekit/file.hpp"
#include "anosecurekit/random.hpp"

anosecurekit::key256 key = anosecurekit::generate_key();

anosecurekit::seal_file("plain.bin", "plain.bin.skf", key);
anosecurekit::open_file("plain.bin.skf", "restored.bin", key);

`SKF1` file shape

  • Fixed 1 MiB chunk size.
  • Random per-file salt.
  • HKDF-SHA256-derived per-file key.
  • AES-256-GCM encryption for each chunk.
  • Chunk nonce made from an 8-byte random file nonce prefix plus a 32-bit chunk index.

Security notes

  • Output paths are not overwritten by path-based file APIs or CLI commands.
  • Wrong keys, wrong AAD, malformed structure, truncation, appended data, and tag failures reject the file.
  • Keep key files out of logs, URLs, command history, backups, crash dumps, and source control.
  • Do not trust stdout or stream output from open operations until the process exits successfully.
  • Use a separate security review before using AnoSecureKit in high-risk systems.
