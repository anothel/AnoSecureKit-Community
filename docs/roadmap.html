Now

The internal backend seam, OpenSSL provider isolation, and build-tree-only external provider hook are verified. The active step is preparing and validating the Community v0.4.0 release while preserving the public API, CLI, packages, and SKT1/SKF1/SKP1 behavior.

External handoff

External security review and package-manager publication remain handoff work until accepted findings or verified package-channel updates exist.

Fix queue

Accepted external-review findings should become one protected change each, with a focused regression check before implementation.

Roadmap rules

  • Security, data loss, and format compatibility come first.
  • Keep OpenSSL as the only shipped Community production provider; a generic internal extension hook does not make an external provider a supported Community backend.
  • Keep CMake install/export, the CLI, and package checks as release surfaces.
  • Do not add public API, wire formats, package channels, CI cost, or release ceremony without a written problem, regression check, and rollback plan.
  • Completed items leave the roadmap and belong in Git history or release notes.

Intake rules

Broad audit documents must be split before implementation. Accepted work must map to an existing AnoSecureKit API, CLI command, serialized format, CMake package surface, release asset, or security-reporting surface.

Put work in `Now` only when it has a named surface, concrete problem, runnable check, and rollback path.

Release-impacting work

Release-impacting changes must pass the configured release preflight:

cmake --build build --config Release --target release-preflight

Before tagging a release, verify package archives, source archives, `SHA256SUMS.txt`, release SPDX SBOM, GitHub artifact attestations, and release notes source of truth.

Out of scope unless the project changes

  • Novel, unpublished, or project-specific crypto primitives, TLS, networking, or secure key storage.
  • Guaranteed memory erasure claims.
  • Framework middleware, JWT, CSRF, CORS, rate limiting, or web adapter parity.
  • Framework-scale abstractions without real call-site pressure.
